# 新手复刻说明

这份说明是给第一次做 Django 项目的同学准备的。

你不用一开始就把所有文件都看完。只要先弄清楚“请求从哪里进来，数据最后存到哪里去”，这个项目就不难。

## 1. 先理解这条主线

用户发一条消息：

```text
commit Tencent Shenzhen Backend 30000
```

系统内部会按下面顺序处理：

1. 微信把消息发到 `/wechat/`
2. `main/views.py` 接收请求
3. `main/lexer.py` 把字符串解析成字典
4. `main/services/offer_services.py` 创建 Offer
5. `main/models.py` 把数据写进 SQLite
6. `main/wechat_utils.py` 生成 XML 回复给微信

只要你理解了这 6 步，这个项目的主干就理解了。

## 2. 每个文件负责什么

## `main/models.py`

这里定义数据库表。

- `User`：记录用户是谁、状态是什么
- `Offer`：记录公司、城市、岗位、薪资，以及对外暴露的 8 位 `public_id`

如果你学过 C++，可以把它理解成“类 + 自动映射到数据库表”。

## `main/services/user_services.py`

这里处理用户相关逻辑：

- 查用户
- 创建用户
- 检查用户状态
- 记录请求频率

这里最重要的点是：项目把“用户状态判断”单独拆出来了，没有直接写死在视图里。

## `main/services/offer_services.py`

这里处理 Offer 相关逻辑：

- 创建一条 Offer
- 批量创建 Offer
- 查询/分页查询（支持按时间/按薪资排序）
- 按 `public_id` 查询单条 Offer（`detail <id>`）
- 编辑与删除（`edit/delete`，且只能操作自己提交的 Offer）
- 最新列表与统计（`latest/stats`）
- “只看我提交的”（`my`）

这是“业务逻辑层”的典型例子。

## `main/lexer.py`

它的作用是把命令字符串变成结构化数据。

例如：

```text
query --city Shenzhen --sort-salary
```

会被解析成类似这样的字典：

```python
{
    "command": "query",
    "city": "Shenzhen",
    "sort-salary": True,
}
```

这一步做完后，后面的代码就不需要再关心原始字符串长什么样了。

现在它还支持解析这些更“像真实产品”的用法：

- `detail 1a2b3c4d`：按 ID 查单条详情
- `edit 1a2b3c4d --salary 32000`：只改某些字段
- `delete 1a2b3c4d` / `delete --all`：删除
- `my --page 1`：只看我提交的
- `latest 5`：看全库最新 N 条
- `stats`：看我提交的统计
- `ping`：连通性测试
- `help query`：只看某个命令的说明

## `main/views.py`

这是请求入口。

它主要做 3 件事：

1. 判断是 `GET` 还是 `POST`
2. 判断是文本消息还是图片消息
3. 把已经解析好的命令交给服务层处理

你可以把它理解成“总调度室”。

## `main/wechat_utils.py`

这里放的是和微信协议更相关的工具函数：

- 生成文本 XML 回复
- 生成图片 XML 回复
- 处理微信首次验证
- 获取 access token

把这些函数独立出来后，`views.py` 就会更干净。

## `main/constants.py`

这里集中放“用户会看到的固定文本”和一些常量。

- 总帮助：`MESSAGE_HELP`（对应 `help`）
- 单命令帮助：`HELP_BY_COMMAND`（对应 `help <command>`）

## 3. 为什么要分层

很多新手第一次做项目时，容易把所有逻辑都写进一个文件。

这样短期看起来快，长期会非常难改。

这个项目用了一个很朴素的分层思路：

- `views.py`：接请求
- `services/`：写业务逻辑
- `models.py`：管数据库

这样分开后，每层只做一件事，阅读起来会轻松很多。

## 4. 为什么把 `eval` 改成 JSON

早期写法会把请求时间队列直接存成字符串，然后再用 `eval()` 转回列表。

这个项目改成了：

- 保存时用 `json.dumps`
- 读取时用 `json.loads`

原因很简单：

- 更安全
- 更容易解释
- 更符合“字符串里存结构化数据”的常见写法

## 5. 你应该先动手改哪里

如果你想自己练习扩展功能，推荐顺序如下：

### 练习 1：改帮助文本

改 `main/constants.py` 里的 `MESSAGE_HELP` 或 `HELP_BY_COMMAND`。

这是最安全的练习，不容易改坏程序。

### 练习 2：新增一个最简单的命令

比如新增：

```text
ping2
```

你至少要改：

- `main/lexer.py`
- `main/views.py`
- `main/constants.py`（如果要写进 help）

这个练习能帮你理解“命令解析 → 分发 → 回复”的完整链路。

### 练习 3：给 `query` 增加一个新筛选条件

比如新增 `--min-salary` 或 `--max-salary` 这类条件。

你至少要改：

- `main/lexer.py`
- `main/services/offer_services.py`
- `main/models.py`（如果这个条件对应新字段）

## 6. 如何判断自己真的看懂了

你可以检查自己能不能回答下面几个问题：

1. 一条 `commit` 命令先经过哪个文件？
2. 数据最终写到哪个模型里？
3. 为什么 `views.py` 不直接写数据库查询？
4. `group-commit` 为什么要检查参数个数是不是 4 的倍数？
5. 为什么用户请求频率检测放在 `user_services.py` 而不是 `lexer.py`？

如果你能回答这些问题，说明你已经不是“只会运行代码”，而是真的开始理解这个项目了。

## 7. 最后建议

- 先跑通，再优化
- 先理解主线，再看细节
- 先改小地方，再改结构
- 每改一次，就重新运行 `python manage.py test`

第一次做项目最重要的不是“写得多高级”，而是“自己能看懂，过几天回来还能继续改”。
